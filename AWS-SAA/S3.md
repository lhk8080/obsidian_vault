> AWS의 **객체 스토리지** 서비스
- 무한하게 스케일링, versioning 지원
- AWS 서비스 중 통합을 위해 S3를 같이 사용하는 경우 (ex. cloudformation)
- Bucket이라는 최상위 저장 공간 (리전 레벨, 정책 설정할 수 있는 논리적 저장 단위)
- 백업 / 스토리지 / DR / 아카이빙 / 하이브리드 클라우드 스토리지 / 어플리케이션 호스팅

### Object
- Key - Value형태 > 모든 파일은 Key를 갖는다
- Object는 prefix + object name으로 구성, 접근
- Bucket에서는 디렉토리라는 개념은 X
- 최대 50TB 크기, 그 이상일 경우 multi part upload
- MetaData / Key / Version ID

### S3 Security
1. S3 접근 통제 
	- User Based : IAM Policy
	- Resource Based : Bucket Policy
		- Object Access Control List
		- Bucket Access Control List
2. 암호화 - 키를 사용하여 객체를 암호화

(+) bucket을 외부에 공개하고 싶지 않다면, Block public access를 반드시 설정해 놓을 것
(+) 오브젝트 외부 공개 >  Block Public Access 해제 + Bucket Policy로 공개 권한 부여

#### Static Website Hosting
- S3는 정적 웹호스팅을 지원
- Bucket Policy > Public Read 설정 필요

#### S3 Versioning
- 객체가 수정되더라도 이전 버전을 보관하여 복구 가능
- 같은 키의 파일 수정되더라도, 덮어쓰지 않고 새로운 버전을 생성
- 버전마다 저장 공간을 사용, 비용 증가
- 버저닝이 중단되어도 이전 버전은 삭제되지 않음

#### S3 Replication
- CRR : Cross Region Replication / SRR : Same Region Replication
- 서로 다른 계정 간 복제 가능 / 비동기 복제
- 복제를 위해서는 IAM permission이 필요
- 기본적으로 Replication 기능을 활성화 하면 새로운 객체에 대해서만 복제가 됨
- 기존 객체까지 복제하려면 S3 Batch Replication 사용
- chaining replication X (A -> B, B -> C 형태의 복제 허용 X)
- buckets > management > replication rules 설정

### S3 Storage Class
> 데이터의 접근 빈도, 복구 속도, 비용에 따라 저장 방식 선택
- Lifecycle 설정을 통해 스토리지 클래스 간 오브젝트를 자동으로 옮길 수 있음(수동도 가능)

- Standard - 낮은 레이턴시, 높은 쓰루풋, 범용
- Infrequent Access (IA) - Standard에 비해 가용성 떨어짐, 가끔 접근, 
	- Standard IA
	- One Zone IA
- Glacier 계열 - 적은 저장 비용 + 객체 검색에 대한 비용
	- Instant Retrieval - 즉시 조회 가능
	- Flexible Retrieval - Expedited / Standard / Bulk
	- Deep Archieve - Standard / Standard
- Intelligent Tiering - S3 접근 패턴 모니터링을 통해 적절한 클래스로 이동
	- 모니터링 + 자동 계층화 비용 청구


#### S3 Durability / Availability
- Duriability : 데이터가 사라지지 않을 확률
	- AWS의 S3 Duriability는 11 nines (99.9999...)
	- 디스크 고장 / 소프트웨어 버그 / 운영 실수 등으로 데이터 손실 가능성 염두ㅗ
- Availability : 서비스 접속할 수 있는 확률
	- S3 Availiability는 99.99%, 1년 中 약 53분 서비스 중단 가능성

(+) S3 Express One Zone
- 단일 AZ에 데이터 저장, 낮은 지연시간과 처리량 > S3 Standard보다 훨씬 빠른 접근

#### S3 Lifecycle Rules

- Expiration Actions

(+) S3 Analytics가 Lifecycle 설정에 도움이 되는 .csv 리포트를 생성


#### S3 Requester Pays
- 기본적으로 S3에 대한 청구는 저장 + 네트워킹 비용

#### S3 Event Notifications
> 버킷에 특정 이벤트가 발생했을 때, 다른 AWS 서비스에 알림을 보냄
- SNS / SQS / Lambda > Resource Access Policy 필요
(+) AWS EventBridge를 통해서 이벤트 통합 가능

#### S3 Performance
- Multi-Part Upload : 분할해서 병렬로 업로드,100메가 파일 이상 권장
- Transfer Acceleration : 전송 속도 높임, 먼저 전 세계 Edge에서 받고, AWS 백본망을 통해 S3 버킷으로 전달 (여러 리전의 사용자가 S3에 접근할 때)
- S3 Byte Range Fetches : 다운로드 시 파일 전체가 아니라 부분(Byte Range)만 가져오기 (대용량 파일 처리, 병렬 다운로드)

#### S3 Batch Operation
> S3에 저장된 객체에 대해 한 번에 작업 수행 (batch)
- 메타데이터/속성 변경, 오브젝트에 대한 암호화, ACL/Tag 변경, Glacier 복원 등의 작업
- S3 Inventory / Athena / Batch Operations

#### Storage Lens
> S3 사용 현황 분석/최적화 해주는 스토리지 분석 서비스 (비용 절감, 데이터 보호)
- 조직 / 계정 / 리전 / 버킷 및 설정 값 등을 고려하여 분석 및 리포트
- 대시보드 > 여러 계정 / 리전에 걸쳐서 데이터를 보여줌
- 여러 메트릭 값이 존재 (cost, data-protection, access-management, event, performance, activity)
- 무료 / 유료 메트릭이 존재

#### MFA Delete
- MFA(Multi Factor Authentication)을 사용하여 S3의 중요 작업 전 추가 인증 작업을 진행
- 오브젝트를 영구 삭제 / 버저닝 중단 등을 진행할 때
- MFA를 사용하려면 Versioning이 활성화되어야 함
- 버킷 소유자만 MFA 활성화/비활성화 할 수 있음
- AWS 콘솔에서는 S3 MFA Delete 활성화 불가, CLI를 통해 루트 계정 로그인 후 MFA 활성화

### S3 Object Encryption
- Server-Side 암호화 (SSE)
	- SSE-S3 : AWS가 키를 관리, default값
	- SSE-KMS : AWS KMS(Key Management Service)로 키를 관리
		- KMS 키에 대한 호출 권한/ KMS API 호출 비용
		- KMS는 TPS 제한이 있음 > 병목 지점 가능성
	- SSE-C : 암호화는 S3가 수행, but 키는 고객이 제공
		- 클라이언트가 키 + 파일 전달
		- 반드시 HTTPS 사용
- Client-Side 암호화 (CSE)

(+) 암호화 강제 : bucket policy 설정
(+) DSSE-KMS : KMS 기반 이중 암후화

### CORS
> Cross Origin Resource Sharing
- 브라우저가 다른 Origin 서버의 요청을 제어하는 보안 정책
- Origin = Protocol + Domain + Port
- SOP, Same-Origin Policy, 브라우저는 기본적으로 다른 오리진 요청 응답을 읽지 않음
- 크로스 오리진의 응답 자체를 막는 것이 아니라, 응답 -> 자바스크립트 전달을 브라우저 단에서 막음(fetch)
- 서버에서 응답에 Access-Control-Allow- (허용할 오리진, 메소드 등)을 추가해줘야 함
- S3 bucket > Permission > CORS 설정

(+) Host : 목적지, Origin : 출발지 > CORS는 Origin을 확인
(+) preflight 요청 : Put, Delete, Authorization 등의 조건에서는 preflight 요청


#### S3 Access Logs
> S3에 대한 모든 접근 요청을 다른 S3 버킷에 로그로 기록
- 로그는 성공/실패 여부와 관계 없이 모두 기록
- 로깅을 위한 버킷은 감사 대상 버킷과 반드시 같은 리전에 존재해야 함
- 하나의 버킷을 App / Logging 용으로 동시에 사용할 수 없음 (Loggin Loop)
- 보안 감사(Audit) / 트러블 슈팅 / 비정상적인 접근 패턴 분석

#### Pre-signed URL

- URL을 생성한 유저의 권한을 상속, 제한된 시간동안 객체 업로드/다운로드 가능
- Private으로 운영되는 버킷 > 특정 파일에 대한 외부 임시 접근이 필요할 때
- URL 생성은 S3 관련 권한을 가진 주체만 가능(IAM user/role, ec2, lambda 등)
- 백엔드 서버 부하를 줄일 수 있음 (url을 통해 사용자가 S3에 바로 접근)

#### Glacier Valut Lock / S3 Object Lock
> **특정 기간** 동안 데이터 삭제/변경 불가하도록 락을 걸어놓음

**Glacier Valut Lock**
- S3 Glacier쪽 옵션, 규제 준수를 위해 사용 (실수/악의적 삭제 방지, root user/AWS도 삭제 불가)
- Write Once Read Many(WORM), 한 번 쓰면 읽기만 가능
- Valut Lock Policy 생성

**S3 Object Lock**
- S3 객체에도 WORM 모델 설정 가능
- versioning이 반드시 활성화 되어 있어야 함
- Retention mode : Complience(엄격하게 관리), Governance(융통성있게 관리, 특별 권한을 가진 경우 삭제/수정 가능)
(+) Legal Hold : retention period와 관계 없이 오브젝트를 관리, legal hold 퍼미션 필요

#### S3 Access Points
> 하나의 버킷에 대해서, 여러 엑세스 포인트로 권한별 S3 접근제어
- 하나의 버킷에 여러 팀/어플리케이션이 같이 접근해야 할 때 사용
- 엑세스 포인트 별로 독립적인 정책 / 고유한 DNS를 가짐
- vpc origin access point - 외부 인터넷을 거치지 않고 EC2 > VPC Endpoint > Access Pont > S3 접근, 특정 vpc만 접근 허용 가능


#### S3 Object Lambda
> S3 객체를 가져올 때, Lambda 함수로 데이터를 가공해서 가져옴
- 여러개의 버킷을 두지 않고, 하나의 버킷 > Lambda로 변형해서 여러 형태의 객체를 사용
- S3 Access Point > Object Lambda
